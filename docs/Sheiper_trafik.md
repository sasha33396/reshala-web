# 🚦 Шейпер трафика Reshala (eBPF + EDT) — Multi-Rule

> **Версия системы:** v4.0.1 (Multi-Rule)
> **Движок:** eBPF + EDT (Earliest Departure Time)
> **Поддержка ядер:** 5.4+
> **Максимум правил:** 32 одновременно

---

## 🧠 Как это работает в целом

Шейпер работает **прямо в ядре Linux** через eBPF-программу, прикреплённую к сетевому интерфейсу. Он перехватывает каждый пакет, определяет какому пользователю (IP) он принадлежит, и выставляет время его отправки (EDT — Earliest Departure Time).

**Ключевые принципы:**

- 🎯 **Per-IP лимит** — каждый уникальный IP получает свой независимый лимит скорости
- 🔢 **Multi-Rule** — одновременно работает до 32 независимых правил с разными настройками
- 🔌 **Per-Port** — каждое правило привязано к своим портам (443, 8443, и т.д.)
- 📡 **DL + UL раздельно** — скачивание и загрузка ограничиваются независимо
- ✅ **0% коллизий** — в отличие от старого `tc u32 hash`, eBPF гарантирует точное соответствие IP → лимит
- 🌐 **IPv6 поддержка** — работает с IPv4 и IPv6 одновременно

## 📐 Архитектура Multi-Rule

```
Входящий пакет
      │
      ▼
 [ Парсим IP + Порт ]
      │
      ▼
 port_rule_map[порт] ──▶ rule_id
      │                  (если порт не найден → пакет проходит без ограничений)
      ▼
 config_map[rule_id] ──▶ { режим, скорости, burst, штраф }
      │
      ▼
  user_state[{IP, rule_id}] ──► Задержка отправки (Download)
                           ├──► Отбрасывание пакетов (Upload)
      │
      ▼
 Пакет уходит или отбрасывается (скорость = лимит)
```

**Три BPF-карты (хранилища в ядре):**

| Карта | Тип | Что хранит |
|---|---|---|
| `port_rule_map` | HASH | порт → rule_id |
| `config_map` | ARRAY | rule_id → настройки правила |
| `user_state_map_down/up` | HASH | {IP + rule_id} → состояние пользователя |

## ⚙️ Режимы работы

### 🔵 Режим 1: Статический (Per-IP)

Жёсткий лимит **всегда**. Пользователь получает ровно N МБ/с в любой момент времени.

```
Пользователь > 0 МБ → EDT выставляется точно
Пользователь >= лимит → EDT выставляется с задержкой
```

**Когда использовать:**
- VPN-сервера (стабильное качество важнее свободы)
- Игровые серверы (нужна предсказуемая задержка)
- Корпоративные сети

**Пример:**
```
Правило #0: Порт 443, DL=5 МБ/с, UL=5 МБ/с, Статика

IP: 1.2.3.4   → всегда ровно 5 МБ/с ✅
IP: 5.6.7.8   → всегда ровно 5 МБ/с ✅
IP: 9.9.9.9   → всегда ровно 5 МБ/с ✅
```

---

### 🟡 Режим 2: Динамический (Burst + Штраф, Per-IP)

Умный режим: пользователь сначала качает быстро, но если превышает квоту — скорость падает.

#### Как работает механизм штрафа:

```
Фаза 1: BUST (нормальная скорость)
  ├── Пользователь качает на полной скорости DL/UL
  ├── Шейпер считает суммарный трафик в окне X секунд
  └── Если скачано > КВОТЫ за окно → Фаза 2

Фаза 2: ШТРАФ
  ├── Скорость падает до штрафной (например 0.5 МБ/с)
  ├── Штраф длится N секунд
  └── После N секунд → обратно в Фазу 1

Штрафная скорость = 0 → полная блокировка на время штрафа
```

#### Параметры динамического режима:

| Параметр | Что означает | Пример |
|---|---|---|
| **DL / UL МБ/с** | Нормальная скорость | 5 МБ/с |
| **Штрафная скорость** | Скорость во время штрафа (0 = блок) | 0.5 МБ/с |
| **Квота (МБайт)** | Сколько можно скачать за окно | 100 МБ |
| **Окно (сек)** | Временной интервал для квоты | 10 сек |
| **Штраф (сек)** | Сколько длится штраф | 60 сек |

#### Пример жизненного цикла пользователя:

```
00:00  Подключился. Скорость: 5 МБ/с
00:08  Скачал 100 МБ за 8 сек → КВОТА ПРЕВЫШЕНА
00:08  Штраф! Скорость: 0.5 МБ/с  (⚡ пользователь замечает торможение)
01:08  Штраф снят. Скорость: 5 МБ/с снова
...
01:15  Снова скачал 100 МБ → снова штраф
```

**Обычный пользователь (соцсети, VoIP, мессенджеры)** — за 10 секунд скачивает < 1 МБ → штрафа **никогда не получает**.

**Качальщик (торренты, loadtest)** — постоянно превышает квоту → сидит в штрафе.

---

### 🟢 Режим 3: Общее ограничение (Shared Pool)

Режим создания "общей трубы". В этом режиме лимит применяется не к каждому IP-адресу по отдельности, а **в сумме ко всем пользователям**, работающим по этому правилу.

```
IP: 1.2.3.4  ┐
IP: 5.6.7.8  ├─► Общий пул: 10 МБ/с
IP: 9.9.9.9  ┘
```

**Когда использовать:**
- Защита основного канала сервера от переполнения (DDoS или гиперактивность).
- Выделение строгой квоты для нестабильных туннелей.
- Если у вас канал на сервере 100 МБ/с, и вы хотите гарантированно оставить 20 МБ/с для SSH и веб-сайтов, вы можете ограничить ВЕСЬ VPN-трафик до 80 МБ/с.

**Особенности:**
- Скорость делится динамически. Если в трубе 100 МБ/с сидит 1 человек, он качает 100. Если сидит 3 человека, они качают примерно по 33 МБ/с.
- eBPF достигает этого за счёт объединения всех пакетов порта в один единый EDT-контейнер, полностью игнорируя IP-адрес отправителя/получателя.

## 📋 Multi-Rule: несколько правил одновременно

С версии v4.0 можно настроить **разные лимиты на разные порты**.

### Пример конфигурации VPN-сервера:

```
Правило #0  │  Порт: 443    │  Статика  │  DL/UL 5 МБ/с
Правило #1  │  Порт: 8443   │  Динамика │  DL/UL 3 МБ/с, Квота 100 МБ/10с, Штраф 0.5 МБ/с 60с
Правило #2  │  Порт: 2083   │  Статика  │  DL/UL 10 МБ/с (премиум-пользователи)
```

**Что это значит:**
- Пользователь на порту 443 → получает 5 МБ/с жёстко
- Пользователь на порту 8443 → может получить 3 МБ/с, но с защитой от качальщиков
- Пользователь на порту 2083 → получает 10 МБ/с (другой тариф)

**Каждое правило:**
- Хранит собственный список портов
- Имеет собственный режим (статика / динамика независимо)
- Ведёт собственную статистику по пользователям

---

### 🥇 Приоритетная маршрутизация и Fallback (Порт `0`)

Шейпер работает по принципу **строгого перехвата (Priority Routing)**. Это значит, что точные порты всегда имеют приоритет над правилом "ВСЕ ПОРТЫ". 

**Киллер-фича (Пример VIP-тарифов):**
Вы можете легко разделять обычных и премиум клиентов без ущерба друг другу на одном сервере.

1. **Создайте Правило #0 (Wildcard / Стандарт):**
   - Выберите порты: `0` (ВСЕ ПОРТЫ)
   - Режим: Статика или Общее ограничение
   - Скорость: `1 МБ/с` (Жесткий лимит для всех нестандартных подключений)
2. **Создайте Правило #1 (VIP):**
   - Выберите порт: `443` (или любой другой, который вы выдаете VIP-клиентам)
   - Скорость: `5 МБ/с`

**Как это сработает:**
Когда клиент подключается на порт `443`, eBPF видит точное совпадение и **не пускает** его в "мусорную корзину" (Правило 0). VIP-клиент получает свои чистые 5 МБ/с. А вот если на сервер придет трафик по любому другому порту (например, `8080`), eBPF не найдет для него точного правила и автоматически сбросит его в Правило #0 ("ВСЕ ПОРТЫ"), задушив до 1 МБ/с. 

*Примечание: в статистике активные IP будут отображаться только в том правиле, через которое реально прошёл трафик.*

## 🗂️ Управление правилами через меню

```
🚦 Шейпер трафика (eBPF + EDT)
│
├── [1] 📋 Активные правила         — просмотр всех настроенных правил
├── [2] 📊 Статистика               — топ IP по правилам, кто в штрафе
├── [3] ➕ Добавить / изменить       — wizard: шаг 0..4
├── [4] 🗑  Удалить правило           — удалить по ID
├── [5] 🧹 Полная очистка           — сброс всех правил
├── [6] 📜 Лог сервиса              — journalctl
├── [7] 🔄 Перезапустить движок     — пересборка + рестарт
└── [8] 📈 Мониторинг (iftop)       — живой трафик в МБ/с
```

### Wizard добавления правила (шаги):

```
Шаг 0: ID правила (0..31)
  └── Показывает текущие правила, предлагает свободный номер

Шаг 1: Сетевой интерфейс
  └── Только при первом запуске движка
  └── Повторные правила — пропускают этот шаг

Шаг 2: Режим
  └── [1] Статический
  └── [2] Динамический
  └── [3] Общее ограничение (Shared Pool)

Шаг 3: Порты
  └── Одиночный: 443
  └── Несколько: 443,8443,2083
  └── При конфликте: предупреждение + вопрос

Шаг 4: Скорости
  └── DL / UL: дробные числа (0.5, 1.5, 3, ...)
  └── Для Динамики: штрафная скорость, квота, окно, длительность штрафа
```

## 🔒 Конфликт портов

Если ты добавляешь правило с портом, который уже используется в другом правиле — система предупредит:

```
⚠️  Некоторые порты уже заняты другими правилами:

  Порт 443 → Правило #0: режим=Статика, порты=[443], DL=5 МБ/с, UL=5 МБ/с

  Переназначить эти порты на новое правило? [y/N]:
```

## 💾 Хранение конфигурации

Все правила **автоматически сохраняются** в файл:
```
/etc/reshala/traffic_limiter/rules.json
```

При рестарте сервиса (или перезагрузке сервера) правила **автоматически восстанавливаются** без участия пользователя.

```bash
# Systemd при старте вызывает:
reshala_ctrl.py restore
# → читает rules.json → применяет все правила в BPF-карты
```

## 🛠️ Ручное управление (CLI)

Контроллер `reshala_ctrl.py` можно запускать прямо из командной строки:

```bash
# Посмотреть все правила
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps rules

# Добавить правило
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps \
  set --rule-id 0 --mode 1 --ports 443 --down 5 --up 5

# Добавить динамическое правило
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps \
  set --rule-id 1 --mode 2 --ports 8443 \
      --down 3 --up 3 --pen 0.5 --burst 100 --win 10 --pen-sec 60

# Добавить "Общее ограничение" (Shared)
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps \
  set --rule-id 2 --mode 3 --ports 2083 --down 100 --up 100

# Удалить правило
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps \
  delete --rule-id 1

# Статистика (топ-10)
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps status

# Статистика по конкретному правилу
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps status --rule-id 0

# Полный список всех IP
python3 /opt/reshala/modules/local/reshala_ctrl.py \
  --pin-dir /sys/fs/bpf/reshala/maps status --full
```

## ⚡ Сравнение с предыдущей системой (tc + u32 hash)

| Характеристика | Старая система (tc u32) | Новая система (eBPF EDT) |
|---|---|---|
| Движок | `tc htb` + `u32 hash` | eBPF EDT (Download) + Token Bucket (Upload) |
| Коллизии IP | ❌ ~63% при 256+ юзерах | ✅ 0% (точный hash-map) |
| Производительность | ❌ Падает при 1+ Гбит/с | ✅ 10+ Гбит/с (Без узких мест ifb0) |
| IPv6 | ❌ Отдельная настройка | ✅ Из коробки |
| Несколько правил | ❌ Один набор настроек | ✅ До 32 правил |
| Динамический режим | ❌ Нет | ✅ Burst + Штраф |
| Дробные скорости | ❌ Только целые Мбит/с | ✅ 0.5 МБ/с и т.д. |
| Статистика по IP | ❌ Нет | ✅ Полная + per-rule |
